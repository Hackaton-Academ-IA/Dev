Décomposition : Landing Page — Server vs Client Components

Next.js App Router — chaque nœud vert est un Server Component (RSC), jaune = Client Component ('use client'). La question clé : où placer la frontière?

■ Server Component ■ Client Component
src/app/
page.tsx— Landing Page (RSC, no state)
layout.tsx— fonts, metadata
src/components/landing/
NavBar.tsx ← onClick CTA + mobile menu state
LogoPixel.tsx— pixel art static
Hero.tsx ← static shell, passe CTAButton dedans
CTAButton.tsx← onClick handler
Tower.tsx← CSS animation only (scrollUp, climb)
ConceptSection.tsx ← 100% statique
LoopNode.tsx
LoopArrow.tsx
TargetCards.tsx ← statique
MarqueeSection.tsx ← scrolling marquee (CSS anim, mais besoin de pause on hover)
StatsStrip.tsx
FAQSection.tsx
src/components/ui/
PixelIcons.tsx ← SVG purs, aucun state → Server

3 stratégies de frontière Client/Server

A

Maximum RSC ● Recommandé

Seuls CTAButton, NavBar (mobile menu) et MarqueeSection sont Client. Tower, Hero, ConceptSection restent Server.

Avantages

  • Bundle JS minimal
  • SEO optimal
  • Streaming SSR

Inconvénients

  • Prop drilling via Server→Client boundary (useRouter dans CTAButton)
B

Hero Client partiel

Hero.tsx est Client (simplifie la propagation du callback onCTA vers NavBar + CTA). Reste Server : ConceptSection, Tower, etc.

Avantages

  • Migration plus directe depuis le HTML
  • Moins de fichiers

Inconvénients

  • +~4KB bundle (Hero inclut Tower)
C

Page Client entière

'use client' sur page.tsx. Migration directe 1-pour-1 depuis le HTML source.

Avantages

  • Le plus rapide à écrire

Inconvénients

  • Pas de SEO
  • Bundle lourd
  • Mauvaise pratique App Router
